# 遗言 / LastWords

### 想换一个 Claude Code 会话,但不想从头讲起。

---

在 Claude Code 里做长任务,你迟早会遇到这三种情况之一:

**1. 会话太贵了。** Opus 4.7 跑到 700k、800k,模型还在正常工作,但每问一句都在把整个上下文重新塞一遍。再跑几轮,今天这点预算就没了。理性选择是新开一个会话。

**2. 模型开始忘前文了。** 上下文还没满,但你能明显感觉到它在胡说——刚确认过的事它又问一遍,刚排除的方向它又提一次,回答越来越水。再聊下去是浪费时间。

**3. 中转站签名 / 历史记录不稳。** 技术层面直接崩了,不是你想不想继续的问题,是已经没法继续了。

这三种情况,表面上是三种不同的失败,本质上是同一个困境:

**你想换会话,但进度没地方放。**

你打开新窗口,光标在空白对话框里闪——

- 刚才我排除了哪些方向?
- 那个冲突我最后是怎么解的?upstream 改了什么、我加了什么?
- 我当时决定不走 A 方案,是因为什么?
- 下一步本来打算做什么来着?

`git log` 没有答案。commit message 没有答案。你自己的笔记也没记这么细——因为当时觉得"反正都在对话里"。

这些判断只存在于那个你已经关掉(或已经不可用)的会话里。

**你丢的不是聊天记录,是工作状态。**

`/遗言` 就是在你关掉那个会话之前(或者趁它还没彻底崩掉),把这些隐性进度显式化、落盘,交给下一个会话。

## 它做什么

在 Claude Code 里输入 `/遗言`(或 `/lastwords`),它会回顾当前会话,整理出:

- **项目背景**——这个 PR / 任务是什么、仓库结构、当前分支
- **已完成的工作**——具体做了什么,精确到 commit hash 和文件
- **未完成的工作**——还差什么,按优先级排
- **关键决策与发现**——为什么这么做、踩过哪些坑、排除了哪些方向
- **下一步建议**——新会话接手后第一步该做什么

然后存成一个 Markdown 文件,丢在当前项目的 `遗言/` 目录下。

新会话 `cat` 一下这份文件,从 0 tokens 开始,30 秒进入状态。

## 和 `/compact` 不是一回事

很多人第一反应是:"这不就是 `/compact` 吗?"——但 `/compact` 解决不了上面任何一个问题。

- **太贵?** compact 完还是在同一个会话里,token 该烧还是烧,只是暂时腾出点空间。
- **模型在忘?** compact 本身也是模型做的,它记不住的东西,压缩出来的摘要只会更糊。
- **签名崩了?** compact 需要会话还能正常工作,崩了根本调不出来。

|  | `/compact` | `/遗言` |
|---|---|---|
| 解决的问题 | 当前会话上下文太挤 | 当前会话要关了,进度怎么办 |
| 操作 | 压缩,继续聊 | 落盘,关窗口 |
| 下一轮成本 | 还在原 context 基础上烧 | 新会话从 0 开始 |
| 产出位置 | 留在会话内存里 | Markdown 文件 |
| 结构 | 自动摘要,不可控 | 背景 / 进度 / 决策 / 下一步 |

一句话:**`/compact` 是续命,`/遗言` 是换命。**

还想在原会话里继续撑一下,用 `/compact`。决定换新会话,用 `/遗言`。两个不冲突——常见用法是先 `/compact` 抢救一下,发现救不回来,再 `/遗言` 落盘走人。

## 文件名就是 TL;DR

普通 handoff 文件长这样:

```
handoff_2026_04_23_143052.md
```

看不出里面是啥。打开之前得先猜。

`/遗言` 生成的长这样:

```
遗言/PR里63个commit残骸squash完了但旧分支夹带货还没从changed_files剥干净-143052_23042026.md
遗言/明明只差attention层的fp16对齐就能跑通了-091523_23042026.md
遗言/ResNet推理结果和PyTorch差了0.3%我真的会谢-220147_23042026.md
```

不用打开,看文件名就知道:什么任务、卡在哪一步、值不值得现在接着干。

轻小说标题不是搞笑,是压缩——把上下文最关键的一句塞进文件名。当你有十几个未完成任务堆在 `遗言/` 目录下,这个压缩就是生产力。

## 安装

一键:

```bash
curl -fsSL https://raw.githubusercontent.com/zuiho-kai/claude-lastwords/master/install.sh | bash
```

用 gh:

```bash
gh repo clone zuiho-kai/claude-lastwords /tmp/lastwords && bash /tmp/lastwords/install.sh
```

手动(两个文件丢进 `~/.claude/commands/` 就行):

```bash
mkdir -p ~/.claude/commands
curl -fsSL https://raw.githubusercontent.com/zuiho-kai/claude-lastwords/master/遗言.md -o ~/.claude/commands/遗言.md
curl -fsSL https://raw.githubusercontent.com/zuiho-kai/claude-lastwords/master/lastwords.md -o ~/.claude/commands/lastwords.md
```

## 使用

在当前会话里(趁它还清醒、还没贵到离谱、还没彻底崩):

```
/遗言
```

或者英文输入法下:

```
/lastwords
```

生成的文件在当前项目的 `遗言/{标题}-{时间戳}.md`。

关掉这个会话,新开一个:

```
请阅读这份遗言文件,了解之前会话的背景和进度,然后继续未完成的工作:
cat 遗言/PR里63个commit残骸squash完了但旧分支夹带货还没从changed_files剥干净-143052_23042026.md
```

从 0 tokens 开始,30 秒后你又回到了原来的工作状态。

## 什么时候该用

判断标准只有一条:

> 如果这个会话现在关掉,我明天能不能接上?

能接上,不用。接不上——哪怕只是有点悬——就 `/遗言`。

三个最高频的触发点在开头已经说了(贵 / 忘 / 崩),其他情况:

- 复杂 PR 做到 squash / rebase / scope 清理阶段,换会话风险高
- 调试做了一半,今天得先停,明天继续
- 要换终端 / 换机器继续干

如果你的任务通常几轮就结束,这东西对你没用。如果你经常"思路还在,但会话该换了",它就是为你做的。
